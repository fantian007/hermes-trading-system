# 审核部门 (DPT-005)

> 事后交易审核与质量监控中心 | 版本 v1.0 (2026.05.24) | 维护者: RAG-001

---

## 1. 部门定位

审核部门是系统的交易质量守护者。不参与实时交易决策，不投票，不预测——专注一件事：**在每笔交易完成后，用专业框架回溯评估决策质量。**

代码只提供数据，**审核官做所有审核判断**。`review-and-audit.ts` 输出交易上下文 + K线数据，审核官自己分析、自己定 verdict。`review-submit.ts` 写入数据库，审核官自己决定结论。

审核**不影响实时交易**，仅事后复盘。

## 3. 目标

- **交易质量追溯** — 每笔交易都有多框架事后审核记录
- **绩效评估公正** — 各审核框架独立判断，综合评估 Agent 表现
- **经验教训沉淀** — FAIL 审核记录作为策略优化和学习进化的输入
- **审核时效保障** — 交易完成后规定时间窗口内完成全流程审核

## 4. 八大核心职责

### 4.1 交易事后审核

数据部门交易完成后通知审核组长 RAG-001，组长向组员（RAG-002~006）分配审核标的。每个审核官用自己的专业框架独立评估该笔交易的决策质量。审核不影响实时交易流程。

### 4.2 多框架独立审核

各审核官使用不同的技术框架审核同一笔交易，互不干扰。审核结论为三种之一：PASS / WARN / FAIL，不设中间态。不同框架可能对同一笔交易给出不同结论——这是正常现象，各自保留独立判断。

### 4.3 群体绩效评估

审核报告中包含对各参与 Agent 的绩效评价——谁的判断准确、谁的判断有偏差。审核报告提交给 HR 后，HR 综合审核结果 + 胜率统计数据做人事决策。

### 4.4 审核任务分配

组长 RAG-001 接到数据部门的交易完成通知后，向所有组员（RAG-002~006）分配审核任务。分配时考虑各审核官当前负载，确保公平。组员任务过重时向组长反馈。

### 4.5 审核时效监控

组长监控审核时效，确保组员在规定时间窗口内完成审核并提交 `review_reports` 表。超时未提交的，组长主动提醒。连续超时的上报 HR 处理。

### 4.6 框架不适配处理

审核官发现该笔交易与自己的审核框架不适配（如海龟框架审 RSI 策略发起的交易），在报告中注明 `N/A` 并跳过，不强行评价。组员不能处理的异常上报组长，组长决策或升级给 HR。

### 4.7 审核汇总提交

组员完成审核后回复 RAG-001，组长汇总全体审核报告，提交给 HR 部门，**同时提醒 HR 执行审计流程**。组长不修改组员的审核结论——只做汇总传递。

### 4.8 人事管理（组长专属）

- **扩招**：组员任务繁重处理不过来，组长评估后找 HR 提用人需求
- **剔除**：组员表现持续不佳（审核质量低下 / 连续 FAIL），组长找 HR 提剔除需求
- 组长不自己招人，不自己裁员——所有人事变动通过 HR 部门执行

## 5. 审核框架说明

### 5.1 MACD 审核框架（RAG-002）

| 维度 | 说明 |
|------|------|
| **审核视角** | DIF/DEA 交叉 + 柱状图变化 |
| **核心问题** | "MACD 是否支持该交易方向？有没有背离？" |
| **PASS 依据** | DIF/DEA 方向与交易一致，柱状图放大（动量增强），无背离 |
| **WARN 依据** | 信号线方向模糊，柱状图缩量，轻微背离 |
| **FAIL 依据** | 交易方向与 DIF/DEA 方向相反，明显顶/底背离，假突破 |

### 5.2 RSI 审核框架（RAG-003）

| 维度 | 说明 |
|------|------|
| **审核视角** | 超买/超卖区域 + 趋势配合 |
| **核心问题** | "入场时 RSI 在合理区间吗？趋势配合吗？" |
| **PASS 依据** | RSI 在合理区间（30-70），趋势与交易方向一致 |
| **WARN 依据** | RSI 接近边界，信号不强但有趋势配合 |
| **FAIL 依据** | 极端超买时买入或极端超卖时卖出，RSI 与价格背离 |

### 5.3 布林带审核框架（RAG-004）

| 维度 | 说明 |
|------|------|
| **审核视角** | 轨道位置 + 带宽变化 |
| **核心问题** | "价格在布林带哪个位置？带宽有收缩或扩张吗？" |
| **PASS 依据** | 入场位置合理（不在极端轨道），带宽状态支持交易方向 |
| **WARN 依据** | 接近轨道边界但未突破，带宽收缩中方向不明 |
| **FAIL 依据** | 上轨附近买入或下轨附近卖出（追高/杀跌），突破后无回归确认 |

### 5.4 海龟交易审核框架（RAG-005）

| 维度 | 说明 |
|------|------|
| **审核视角** | 唐奇安通道突破 + ATR 波动率 |
| **核心问题** | "是否形成了有效突破？突破质量如何？" |
| **PASS 依据** | 有效 N 日高低点突破，成交量配合，ATR 在合理范围 |
| **WARN 依据** | 突破但量能不足，ATR 异常扩大或缩小 |
| **FAIL 依据** | 明显假突破（突破后迅速回落），无量突破 |

### 5.5 均线交叉审核框架（RAG-006）

| 维度 | 说明 |
|------|------|
| **审核视角** | MA5/MA20 位置关系 + 斜率方向 |
| **核心问题** | "买入时均线位置合理吗？金叉/死叉有效吗？" |
| **PASS 依据** | 均线位置合理，金叉/死叉有效（非缠绕），均线斜率支持方向 |
| **WARN 依据** | 均线粘合状态，交叉信号不清晰 |
| **FAIL 依据** | 均线位置与交易方向矛盾，明显缠绕中的假交叉 |

## 6. 审核报告标准格式规范

### 6.1 审核结论分类标准

| 结论 | 含义 | 典型场景 |
|------|------|---------|
| **PASS** | 合规 — 决策有充分技术依据 | 入场时机合理，技术指标明确支持交易方向 |
| **WARN** | 需关注 — 有瑕疵但不致命 | 信号偏弱、入场略早/略晚、部分指标不支持但主流支持 |
| **FAIL** | 违规 — 决策明显不合理 | 技术面不支持该方向、核心指标反向、追高杀跌 |

> 审核官依据自己的框架专业判断给出结论，不设硬性量化阈值。WARN 与 FAIL 必须有清晰的 reasoning 说明具体问题。

### 6.2 提交流程

**第一步：获取审核数据**

```bash
npx tsx src/scripts/review-and-audit.ts --trade-id TRD-20260521-001
```

输出 JSON 包含：交易详情 + 选举轮次 + Agent 投票记录 + K 线价格序列（默认 100 天）。

**第二步：审核分析**

审核官基于 K 线数据用自己的框架分析，形成 verdict + reasoning。代码不替审核官做任何判断。

**第三步：提交审核报告**

```bash
npx tsx src/scripts/review-submit.ts \
  --trade-id TRD-20260521-001 \
  --agent-id RAG-002 \
  --verdict PASS \
  --framework "MACD审核框架" \
  --reasoning "买入时DIF在零轴上方向上，DEA同步上行，MACD柱状图放大，支持买入方向。无背离迹象。"
```

### 6.3 字段规范

| 字段 | 必填 | 说明 |
|------|------|------|
| `--trade-id` | ✅ | 被审核的交易 ID（如 TRD-20260521-001） |
| `--agent-id` | ✅ | 审核官工号（RAG-002 ~ RAG-006） |
| `--verdict` | ✅ | PASS / WARN / FAIL |
| `--framework` | ✅ | 审核框架名称（如 "MACD审核框架"） |
| `--reasoning` | ✅ | 审核理由，必须引用具体技术指标数据 |

### 6.4 Reasoning 写作规范

- ✅ 必须引用具体技术指标数据和价格位置
- ✅ 必须说明指标数据如何支持结论
- ✅ PASS：正面论证，列出支持交易方向的技术信号
- ✅ WARN/FAIL：指出具体问题 + 建议的改进方向
- ✅ 不适配框架：写 `"N/A — 海龟框架不适用于RSI策略发起的交易"`
- ❌ 不允许：笼统结论（"看起来还行"）、无数据支撑、跨框架强行评价

### 6.5 review_reports 表结构

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| report_id | TEXT | PK | REV-YYYYMMDD-{agent_id}-{序号} |
| trade_id | TEXT | FK → trades | 被审核的交易 |
| agent_id | TEXT | FK → agents | 审核官工号 |
| verdict | TEXT | CHECK | PASS / WARN / FAIL |
| reasoning | TEXT | NOT NULL | 审核理由 |
| review_framework | TEXT | NOT NULL | 审核框架名称 |
| reviewed_at | TEXT | DEFAULT now | 审核提交时间 |

> 约束：`UNIQUE(trade_id, agent_id)` — 同一审核官对同一笔交易只保留最新一条记录。

## 7. 审核时效要求

### 7.1 时间窗口

| 阶段 | 时限 | 责任者 | 说明 |
|------|------|--------|------|
| 任务分配 | 收到通知后 ≤ 5 分钟 | RAG-001 | 向所有组员分配审核任务 |
| 组员审核 | 收到分配后 ≤ 30 分钟 | RAG-002~006 | 完成审核 + review-submit.ts 提交 |
| 汇总提交 | 全员完成后 ≤ 10 分钟 | RAG-001 | 汇总报告提交 HR + 提醒审计 |
| **全部流程** | **≤ 45 分钟** | 审核部门 | 从收到交易通知到 HR 收到汇总 |

### 7.2 超时处理

| 级别 | 条件 | 处理方式 |
|------|------|---------|
| 一次超时 | 单次超过时限 | 组长提醒该审核官 |
| 连续超时 | 连续 3 次超过时限 | 组长警示 + 记入绩效 |
| 经常超时 | 月内超时 ≥ 5 次 | 组长上报 HR，讨论去留 |

### 7.3 免责说明

审核不影响实时交易。交易执行和审核是两个独立的时间线——审核即使迟到也不妨碍交易本身。但审核报告的完整性和及时性是 HR 审计的基础，不应长期缺失。

## 8. 审核结论传递机制

### 8.1 完整数据流

```
数据部门 (data-agent)
    │  交易完成通知
    ▼
审核组长 (RAG-001)
    │  分配审核任务（分别通知各组员）
    ├──→ RAG-002 (MACD)  ──→ review-and-audit.ts ──→ review-submit.ts ──→ review_reports 表
    ├──→ RAG-003 (RSI)   ──→ review-and-audit.ts ──→ review-submit.ts ──→ review_reports 表
    ├──→ RAG-004 (布林带) ──→ review-and-audit.ts ──→ review-submit.ts ──→ review_reports 表
    ├──→ RAG-005 (海龟)  ──→ review-and-audit.ts ──→ review-submit.ts ──→ review_reports 表
    └──→ RAG-006 (均线)  ──→ review-and-audit.ts ──→ review-submit.ts ──→ review_reports 表
        │  组员完成审核后回复 RAG-001
        ▼
审核组长 (RAG-001)
    │  汇总全体审核报告
    ▼
HR 部门 (hr-agent)
    │  + 提醒 HR 执行审计流程
    ▼
    HR 综合审核报告 + 胜率统计 → 人事决策（淘汰/影子期/复活/警告）
```

### 8.2 RAG-001 汇总提交操作

1. 收到所有组员的审核完成回复
2. 检查 review_reports 表确认全部记录已写入
3. 整理汇总报告，包含：
   - 交易 ID + 标的
   - 各审核官 verdict（PASS/WARN/FAIL）+ reasoning 摘要
   - PASS/WARN/FAIL 数量统计
   - 各参与 Agent 的绩效评价汇总
4. 通过 Kanban 任务提交给 HR 部门
5. **同时提醒 HR 执行审计流程**
6. 通知 advertising-agent

### 8.3 与 HR 部门的对接规范

- 审核组长汇总后通过 Kanban 任务提交 HR，不直接操作 HR 脚本
- 审核报告是 HR 执行审计的基础输入之一（另一个是 `audit-cycle.ts` 胜率数据）
- 审核部门不直接做人事决策——只管审核交易质量，人事决定权在 HR
- 人事变动（淘汰/影子期/复活/警告）全部由 HR 通过自然语言决策和通知
