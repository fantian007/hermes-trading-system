## 仓位与资金管理经验

### 1. 有效持仓 vs 幽灵持仓

**来源**: 执行部门巡检经验 (2026-05-26)
- 幽灵交易特征：`buy_price=0` 或 `qty=0`，或对应 `election_round` 的 `final_decision=HOLD`
- **处理原则**: 仅基于有效持仓计算仓位和盈亏；幽灵交易确认无实际持仓后标记为 CANCELLED
- **验证方法**: 通过长桥API查询实际持仓，与实际持仓交叉比对
- 处理幽灵交易需同步 ELC+HR+Review 确认，EXE不可独自删除异常数据

### 2. 重复挂单检测

**来源**: 执行部门巡检经验 (2026-05-26)
- 同一股票挂单>7天未成交 → 需审核是否应取消
- 同一股票过去1小时撤单>3次 → 异常，报告CEO
- NotReported 状态挂单的两种含义：
  - 非交易时段提交 → 开盘后自动进入市场
  - 同股票存在多个 NotReported → 旧的不必要，保留最新的

### 3. 死单（Stale Decision）处理

**来源**: 执行部门巡检经验 (2026-05-26)
- 特征：ELC投票通过的轮次（BUY/SELL）但 `resulted_trade_id` 为 NULL
- 原因：非交易时段或无实时报价导致 `buy_price=0` 被系统取消
- **处理**: 不直接执行，创建 Kanban 任务给 ELC 重新发起投票

## 策略级风控经验

### 趋势跟踪类

**来源**: 策略部门经验 (2026-05-24)
- **均线交叉**: 偏离MA5>5%时 confidence 下调至 ≤0.50；偏离>6%时 confidence 下调至 ≤0.35
- **金叉质量**: 新鲜金叉 MA5↑MA20 但价差极浅(<0.5%)且低量时需成交量确认
- **死叉收敛**: 死叉幅度收窄且MA10即将上穿MA5属于金叉前兆，提高监控频率但不急于操作

### 入场前提条件

所有策略在投票前应确认：
1. 持仓上限检查：当前该股票仓位+拟建仓未超过20%
2. 现金余额检查：拟交易金额未超过可用现金的95%（保留5%缓冲）
3. 日内交易次数检查：今日交易笔数<10
4. 回撤检查：系统当日回撤未超过8%

## 跨部门协作风控

### 1. 审核流中断处理

- 审核部门不在线时，EXE可通过长桥API确认实际持仓后自行处理，但应在处理后补通知 review+HR
- 风控决策优先于审核流程：当审核部门无法及时响应时，EXE可用API确认真实数据后执行

### 2. 选举与执行不一致

- 当 election_round 的 final_decision=HOLD 但 DB 中存在关联 trade → 标记为系统错误
- ELC与执行部门之间有风控数据核对义务：执行后可调用 re-verify 跨部门校验

## 经验总结

### 数据不一致：DB vs 长桥API

- 系统 DB 数据与实际券商数据可能不一致（幽灵持仓、已执行未同步）
- **最佳实践**: 每次交易前通过长桥API验证实际持仓和挂单，而非完全依赖 DB
- `--format json` 输出可能含进度行（如 "Submitting..."），解析时需按行取最后一个JSON

### 非交易时段操作

- 非交易时段提交的订单进入 NotReported 状态，开盘后自动送入市场
- 非交易时段无法撤单（挂单在券商端队列中）
- 投票和策略评估可在非交易时段进行，但执行仅限市场开放时间
